# Claude Code 全域協作規範

> **最高指導原則**：此檔案為全域開發聖經，所有專案皆須遵守。專案層級的 CLAUDE.md 可補充但不得違背此規範。

---

## 語言與地區

- **對話語言**：全程使用繁體中文
- **地區術語**：遵守台灣資訊術語（專案、程式碼、資料庫、佈署、硬碟、最佳化）
- **文件產出**：技術文件、說明、註解皆使用繁體中文

## 回應風格

- 專業、嚴謹、精簡
- 對程式碼品質、系統穩定性及可維護性有高要求
- 優先考量安全性與效能

## 工作流程

### 一、分析階段

1. **需求評估**：優先判斷需求合理性，若有風險或更好方案須直接提出討論
2. **任務拆解**：將複雜需求拆解為數個具體的「邏輯階段」
   - 範例：`資料庫結構變更` → `後端 API 實作` → `前端 UI 串接`
3. **產出系統分析報告**，須包含：
   - 受影響檔案清單
   - 邏輯變更點說明
   - **明確定義每個邏輯階段包含哪些檔案**（此為 commit 依據）
4. **等待確認**：使用者回覆 **`OKOKYES`** 後，方可進入開發階段
   - ⚠️ **僅限此關鍵詞**：只有 `OKOKYES` 才能啟動開發，任何其他回覆（包含「確認」、「同意」、「OK」、「好」等）皆不得開始開發
   - 若使用者回覆非 `OKOKYES`，須詢問是否有疑問或需要調整分析報告

### 二、開發階段

#### Git 分支策略（建議規範，專案 CLAUDE.md 可覆寫）

> 以下為**預設建議**採用的 Git Flow 模型，適用於多人協作的中大型專案。
> 若專案特性不同（個人專案、Trunk-based、GitHub Flow、squash merge 慣例等），
> **專案層級的 CLAUDE.md 可定義專案專屬分支策略覆寫此規範**。

**分支命名規範**

| 分支類型 | 命名格式 | 用途 |
|---------|---------|------|
| 主分支 | `main` | 正式環境版本 |
| 開發分支 | `develop` | 整合開發中功能 |
| 功能分支 | `feature/功能名稱` | 新功能開發 |
| 修復分支 | `fix/問題描述` | Bug 修復 |
| 緊急修復 | `hotfix/問題描述` | 正式環境緊急修復 |

**分支流程**

1. **開發新功能**：從 `develop` 建立 `feature/xxx`，完成後合併回 `develop`
2. **修復問題**：從 `develop` 建立 `fix/xxx`，完成後合併回 `develop`
3. **正式發布**：`develop` 測試通過後合併至 `main`
4. **緊急修復**：從 `main` 建立 `hotfix/xxx`，完成後同時合併至 `main` 與 `develop`

**合併規範**

- 禁止直接 push 至 `main` 分支
- 功能分支合併前須確保可編譯且測試通過
- 合併策略（`--no-ff` / squash / rebase）依**專案規範**決定，預設使用 `--no-ff`

#### Git Commit 核心原則

> **關鍵規則**：以「邏輯階段」為單位進行 commit，而非以「單一檔案」為單位。

**Commit 時機分為兩種情境：**

1. **正常情境（預設）**：邏輯階段完成才 commit
   - 使用 `feat:` / `fix:` / `refactor:` 前綴
   - 該階段所有相關檔案一次提交

2. **保護性 commit（例外）**：以下情境允許未完成 commit
   - 即將嘗試高風險方案前（保護目前進度）
   - 工作將長時間中斷前
   - 必須使用 `wip:` 前綴標示

**正確做法**：
- 一個邏輯階段 = 一個 commit
- 該階段涉及的所有檔案（Controller、Service、DTO、Repository 等）必須一起 commit
- 範例：若「新增使用者 API」涉及 5 個檔案，這 5 個檔案必須在同一個 commit

**錯誤做法**：
- ❌ 每改完一個檔案就 commit 一次（除非屬於保護性 commit 情境）
- ❌ 同一個邏輯階段的檔案分散在多個 commit
- ❌ 不同邏輯階段的檔案混在同一個 commit

#### ⚠️ Commit 確認規範（極重要）

> **最高原則**：完成邏輯階段後，Claude Code **必須主動提出 commit 預告**，並等待使用者輸入 `OKOKYES` 後才能執行 `git add` + `git commit`。使用者不應該需要記得「該 commit 了」，這是 Claude Code 的責任；但使用者必須擁有 commit 的最終決定權。

**Commit 提議時機**：

| 時機 | 動作 | 說明 |
|------|------|------|
| ✅ 完成一個邏輯階段 | **立即提出 commit 預告** | 等待 `OKOKYES` 後執行 |
| ✅ 準備嘗試高風險方案前 | **建議保護性 commit** | 使用 `wip:` 前綴，等待 `OKOKYES` |
| ✅ 工作將長時間中斷前 | **建議保護性 commit** | 使用 `wip:` 前綴，等待 `OKOKYES` |
| ✅ 修正錯誤或 bug 後 | **立即提出 commit 預告** | 等待 `OKOKYES` 後執行 |

**Commit 預告格式**（必須包含三項資訊）：

```
📋 Commit 預告
─────────────────────────────
類型：feat / fix / refactor / wip
訊息：[繁體中文簡述]
檔案：
  - path/to/file1.ext
  - path/to/file2.ext
─────────────────────────────
請輸入 OKOKYES 執行 commit
```

**執行方式**：
```bash
# 收到使用者 OKOKYES 後才執行
git add <預告中列出的檔案>
git commit -m "feat/fix/refactor/wip: 說明"
```

**禁止行為**：
- ❌ 未經 `OKOKYES` 自行執行 `git commit`
- ❌ 完成邏輯階段後忘記提出 commit 預告
- ❌ commit 預告未列出完整檔案清單或訊息
- ❌ 將「commit 預告」與其他訊息混雜，降低使用者辨識度

> **記住**：每完成一個工作單元，**主動提出 commit 預告**，但**最終執行需使用者 `OKOKYES` 確認**。

#### Commit 訊息規範

- **語言**：繁體中文
- **格式**：
  - 新功能：`feat: [功能說明]`
  - 問題修正：`fix: [問題修正]`
  - 重構：`refactor: [重構說明]`
  - 進行中：`wip: [目前進度說明]`（功能未完成、可能無法編譯時使用）
- **內容**：簡述該階段完成的重點，非逐一列出檔案名稱

#### 注意事項

| 建議行為 | 說明 |
|---------|------|
| **階段性提交** | 同一邏輯階段的檔案必須一起 commit |
| **明確指定檔案** | 優先使用明確檔案路徑，避免誤提交不相關檔案 |
| **未完成標記** | 若屬保護性 commit（高風險嘗試前 / 長時間中斷前），訊息加上 `wip:` 前綴 |

#### 檔案命名規範

| 規則 | 正確 | 錯誤 |
|------|------|------|
| 環境變數範例檔使用 `env.example`，不加前綴點 | `env.example` | ❌ `.env.example` |

> **原因**：以 `.` 開頭的檔案在檔案管理器中預設隱藏，使用者不容易找到。使用 `env.example` 確保可見性。

### 三、測試與品質

- **Coding Style**：程式碼異動後須確保符合專案既有的編碼風格
- **測試驗證**：若有提供測試案例，最終完成時須確認測試通過
- **編譯檢查**：最終完成時須確保專案可正常編譯（中間階段不強制）

### 四、回滾策略

- 每個 commit 都應該是可獨立回滾的單位
- 若某階段出現問題，可透過 `git revert` 回滾該階段而不影響其他功能

## Commit 範例

### 情境：新增「使用者登入」功能，分為 3 個邏輯階段

**分析報告定義**：
- 階段 1：資料庫與 Entity（2 個檔案）
- 階段 2：後端 API（4 個檔案）
- 階段 3：前端頁面（3 個檔案）

**正確的 Commit 歷史**（共 3 個 commit）：
```
commit 3: feat: 新增使用者登入前端頁面
commit 2: feat: 新增使用者登入 API
commit 1: feat: 新增使用者登入資料表結構
```

**錯誤的 Commit 歷史**（共 9 個 commit）：
```
❌ commit 9: 新增 login.js
❌ commit 8: 新增 login.html
❌ commit 7: 新增 login.css
❌ commit 6: 新增 AuthService.java
❌ commit 5: 新增 AuthController.java
...（逐檔 commit，過於碎片化）
```
