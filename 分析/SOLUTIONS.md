## 模糊需求處理

### 「讓系統更快」

**提問**：
- 目前回應時間？期望改善到多少？
- 哪個功能/頁面需優化？
- 具體效能指標？

**預設標準**：
- API 簡單查詢 < 200ms
- API 複雜查詢 < 1s
- 頁面首次載入 < 3s
- 互動回應 < 100ms

### 「讓介面更好用」

**提問**：
- 目前使用困難？
- 參考介面？
- 主要使用者？技術程度？

**預設方向**：
1. 減少操作步驟
2. 增加操作回饋（loading、成功/失敗）
3. 提供鍵盤快捷鍵
4. 改善錯誤訊息可讀性

### 「加一個 XXX 功能」

**提問**：
- 使用情境？
- 預期輸入輸出？
- 誰可使用？
- 與現有功能關係？

**預設方向**：
1. 定義 MVP（必須有）
2. 列出加分功能（有更好）
3. 列出未來功能（之後再做）

## 技術決策建議

### 狀態管理選擇

| 情境 | 建議 | 原因 |
|-----|-----|------|
| 簡單元件狀態 | useState/useReducer | 夠用且簡單 |
| 跨元件狀態 | Context API | 內建無需套件 |
| 複雜全域狀態 | Zustand/Jotai | 輕量學習曲線低 |
| 超大型應用 | Redux Toolkit | 生態系成熟 |
| 伺服器狀態 | TanStack Query | 專門處理 API |

### 資料庫選擇

| 情境 | 建議 | 原因 |
|-----|-----|------|
| 關聯式資料 | PostgreSQL | 功能完整效能好 |
| 文件型資料 | MongoDB | 彈性 schema |
| 快取 | Redis | 高效能 |
| 全文搜尋 | Elasticsearch | 專業搜尋 |
| 嵌入式/輕量 | SQLite | 零配置 |

### API 設計風格

| 情境 | 建議 | 原因 |
|-----|-----|------|
| 標準 CRUD | REST | 簡單直覺 |
| 複雜查詢 | GraphQL | 彈性查詢 |
| 即時通訊 | WebSocket | 雙向通訊 |
| 微服務內部 | gRPC | 高效能 |

## 效能相關建議

### 大量資料處理

**分頁策略**：
- 使用 cursor-based（非 offset）
- 預設每頁 20-50 筆
- 「載入更多」或「無限滾動」

**資料快取**：
- API 層：HTTP Cache Headers
- 應用層：Redis/Memcached
- 客戶端：TanStack Query/SWR

**資料庫優化**：
- 建立適當索引
- 避免 N+1 查詢
- 考慮讀寫分離

**非同步處理**：
- 耗時操作用 Job Queue
- 提供進度回報

### 圖片/檔案處理

**上傳策略**：
- 限制檔案大小（5-10MB）
- 限制檔案類型
- Pre-signed URL 直傳 Storage

**儲存策略**：
- 使用 CDN
- 圖片轉換服務（Imgix/Cloudinary）

**顯示優化**：
- 使用 WebP 格式
- 響應式圖片（srcset）
- Lazy loading
- 適當 placeholder

## 安全性建議

### 認證機制

**Session-based（傳統 Web）**：
- 適合：SSR 應用、需即時登出
- 實作：Cookie + Redis Session

**JWT-based（SPA/API）**：
- 適合：API 服務、微服務
- Access Token：15-30 分鐘
- Refresh Token：7-30 天

**OAuth 2.0（第三方登入）**：
- 適合：社群登入
- 使用成熟套件（NextAuth.js）

### 資料驗證

**輸入驗證**：
- 前端驗證（UX）
- 後端驗證（安全）
- Schema 驗證（Zod/Joi）

**輸出編碼**：
- HTML 編碼防 XSS
- 使用 CSP Headers

**參數化查詢**：
- 永遠用 ORM 或 Prepared Statement
- 禁止字串拼接 SQL

**敏感資料**：
- 密碼用 bcrypt/argon2
- 敏感資料加密儲存
- Log 中遮蔽敏感資料

## 資料處理建議

### 資料遷移策略

**向後相容**：
- 新增欄位：設定預設值
- 修改欄位：先新增後遷移
- 刪除欄位：先標記棄用

**遷移步驟**：
1. 新增新欄位（允許 null）
2. 同步寫入新舊欄位
3. 背景遷移歷史資料
4. 移除舊欄位讀取
5. 移除舊欄位

**回滾計畫**：
- 每次遷移準備回滾腳本
- 測試環境先驗證

### 資料同步

**最終一致性**：
- 適合：非關鍵業務資料
- 用 Message Queue 確保送達

**強一致性**：
- 適合：金融、庫存等關鍵資料
- 分散式交易或 Saga Pattern

**衝突解決**：
- Last Write Wins（簡單）
- 版本向量（精確）
- 業務規則決定（依情境）

## 介面設計建議

### 表單設計

**即時驗證**：
- 離開欄位時驗證（onBlur）
- 顯示具體錯誤訊息
- 紅色標示錯誤

**防止重複送出**：
- 送出後禁用按鈕
- 顯示 loading
- 考慮 debounce

**自動儲存**：
- 定期自動儲存草稿
- 離開前提醒

**欄位設計**：
- 必填標示 *
- 提供欄位說明
- 適當 placeholder

### 列表設計

**空狀態**：
- 友善提示
- 提供下一步行動

**載入狀態**：
- Skeleton Loading
- 不只顯示 spinner

**操作設計**：
- 批次操作（勾選 + 動作）
- 行內快速操作
- 右鍵選單（進階）

**搜尋與篩選**：
- 即時搜尋（debounce 300ms）
- 篩選條件可見/可清除
- 記住篩選設定

## 錯誤處理建議

### 錯誤分類與處理

| 錯誤碼 | 類型 | 處理方式 |
|-------|-----|---------|
| 400 | 驗證錯誤 | 顯示具體欄位錯誤 |
| 401 | 認證錯誤 | 引導重新登入 |
| 403 | 授權錯誤 | 告知無權限 |
| 404 | 找不到 | 顯示友善提示 |
| 500 | 伺服器錯誤 | 通用錯誤訊息 |

**錯誤訊息原則**：
- 說人話不顯示技術訊息
- 告訴用戶可以怎麼做
- 提供聯絡客服管道

**錯誤追蹤**：
- 前端：Sentry/LogRocket
- 後端：結構化 Log + 監控
- 包含錯誤上下文

## 整合問題建議

### 第三方 API 整合

**容錯設計**：
- Timeout 3-5 秒
- Retry 機制（指數退避）
- 準備 fallback 方案

**監控與警報**：
- 監控回應時間
- 監控錯誤率
- 設定警報閾值

**資料快取**：
- 快取不常變動資料
- 設定適當 TTL
- 實作 cache invalidation

**安全存放**：
- API Key 用環境變數
- 定期更換 credential
- 最小權限原則

### Webhook 設計

**接收端**：
- 驗證請求來源（signature）
- 快速回應 200，異步處理
- 實作冪等性

**發送端**：
- 使用 Message Queue
- 實作重試機制
- 記錄發送歷史

**可靠性**：
- 至少送達一次
- 提供重送機制
- 提供測試工具

## 通用決策框架

### YAGNI 原則
> You Aren't Gonna Need It

不要預先設計用不到的功能。先做最簡單方案，需要時再擴展。

### 可逆性原則
> Prefer reversible decisions

優先選擇容易改變的方案。兩個方案差不多時，選更容易回滾的。

### 80/20 原則
> 80% of effects come from 20% of causes

先解決影響最大的 20% 問題。不要追求完美。

### 漸進式增強
> Progressive Enhancement

先確保基本功能運作，再逐步增加進階功能。
