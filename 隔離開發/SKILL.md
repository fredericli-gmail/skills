# 隔離開發 Skill

## Overview

實現「主目錄穩定測試 + Worktree 隔離開發」的協作模式。

**核心理念**：
- 主目錄保持穩定，僅用於測試已完成的功能
- 每個開發任務在獨立的 worktree 中進行
- 透過 push/pull 機制同步變更

## 觸發方式

用戶輸入：`/隔離開發 <需求描述>`

## 工作流程

### Phase 1: 需求分析

> 複用「分析」Skill 的邏輯，但不串接「開發」Skill

#### 1.1 需求理解

收到需求後，立即執行：

1. **複述需求** - 用自己的話重新描述
2. **識別關鍵字** - 提取核心概念、實體、行為
3. **界定範圍** - 明確什麼在/不在範圍內

輸出格式：
```
【需求理解摘要】
核心目標：[一句話描述]
主要功能：[功能清單]
涉及範圍：[影響的模組/系統]
預期產出：[最終交付物]
```

#### 1.2 疑問識別

提出問題，分級輸出：

```
【疑問清單】

🔴 關鍵問題（必須回答才能繼續）：
1. [問題]
   建議方案：[Claude 的建議]

🟡 重要問題（影響設計決策）：
1. [問題]
   建議方案：[建議處理方式]

🟢 確認事項（可使用預設值）：
1. [問題]
   預設方案：[預設處理]
```

#### 1.3 程式碼掃描

執行系統化掃描，輸出：

```
【程式碼掃描報告】

📁 專案架構：
[技術棧與架構描述]

📝 相關檔案：
[需修改]
├── path/file1.ts - [修改原因]
└── path/file2.ts - [修改原因]

[可能受影響]
├── path/related1.ts - [影響原因]
└── path/related2.ts - [影響原因]
```

#### 1.4 執行計畫

產出完整的開發計畫，包含：
- 階段拆解
- 每階段的檔案清單
- 完成標準

```
【執行計畫】

階段 1：[階段名稱]
- 目標：[階段目標]
- 檔案：[檔案清單]
- 完成標準：[如何判斷完成]

階段 2：[階段名稱]
...
```

#### 1.5 等待確認

```
請確認以上分析與執行計畫。
確認無誤請回覆「OKOKYES」，我將建立獨立開發環境。
```

⏸️ **停止點：等待用戶回覆 OKOKYES 後才繼續 Phase 2**

---

### Phase 2: 建立開發環境

用戶確認後，執行以下步驟：

#### 2.1 決定分支名稱

根據需求類型決定分支名稱：
- 新功能：`feature/功能名稱`（英文、kebab-case）
- 問題修復：`fix/問題描述`
- 緊急修復：`hotfix/問題描述`

#### 2.2 建立 Worktree

執行指令：

```bash
# 確保在主目錄
cd /Users/fredericli/Git/Danova/Danova-woms

# 建立 worktree（從 develop 分支）
git worktree add ./sub-danova-worm/<分支名稱> -b <分支名稱> develop
```

**目錄結構**：
```
/Users/fredericli/Git/Danova/Danova-woms/
├── src/                          # 主目錄原有內容
├── sub-danova-worm/              # Worktree 容器目錄
│   └── feature-xxx/              # 本次任務的 worktree
│       ├── src/                  # 完整的專案副本
│       └── TASK.md               # 任務說明檔
```

#### 2.3 產生 TASK.md

在 worktree 根目錄產生 `TASK.md`，內容依照 [TASK_TEMPLATE.md](TASK_TEMPLATE.md) 模板。

#### 2.4 輸出切換指令

```
┌─────────────────────────────────────────────────────────────────┐
│ ✅ 開發環境已建立                                                │
├─────────────────────────────────────────────────────────────────┤
│ Worktree 路徑：./sub-danova-worm/<分支名稱>                      │
│ 分支名稱：<分支名稱>                                             │
│ 基底分支：develop                                                │
│ 任務檔案：TASK.md                                                │
├─────────────────────────────────────────────────────────────────┤
│ 請執行以下指令進入開發環境：                                      │
│                                                                 │
│   cd ./sub-danova-worm/<分支名稱> && claude                      │
│                                                                 │
│ 進入後輸入「開始」即可開始執行任務。                              │
└─────────────────────────────────────────────────────────────────┘
```

## Worktree 內的 Claude 行為

當用戶在 worktree 中啟動 Claude 並輸入「開始」時：

1. **讀取 TASK.md**
2. **依計畫執行開發**
3. **每階段完成後**：
   - 立即 commit（同階段檔案一起）
   - push 到遠端
   - 告知用戶：「階段 X 已完成並 push，您可以在主目錄執行 `git fetch && git checkout <分支>` 測試」

4. **所有階段完成後（強制執行）**：
   - 確認專案可編譯（`./mvnw compile`）
   - 輸出「開發完成通知」
   - 等待用戶輸入「OKOKYES」
   - 執行 `/CodeReview` skill

## 用戶測試流程

用戶在主目錄測試：

```bash
# 回到主目錄
cd /Users/fredericli/Git/Danova/Danova-woms

# 拉取遠端分支
git fetch origin

# 切換到開發分支測試
git checkout <分支名稱>

# 啟動應用程式測試
./mvnw spring-boot:run
```

---

## 清理 Worktree

功能完成並合併後：

```bash
# 回到主目錄
cd /Users/fredericli/Git/Danova/Danova-woms

# 合併到 develop
git checkout develop
git merge <分支名稱>

# 刪除 worktree
git worktree remove ./sub-danova-worm/<分支名稱>

# 刪除本地分支（可選）
git branch -d <分支名稱>
```

---

## 注意事項

1. **sub-danova-worm 目錄**：建議加入 `.gitignore`，避免意外提交
2. **同時開發多個任務**：每個任務一個 worktree，各自獨立
3. **基底分支**：預設從 `develop` 建立，遵循專案的分支策略

---

## 禁止行為

- ❌ 在 Phase 1 完成前就建立 worktree
- ❌ 接受 OKOKYES 以外的確認詞彙
- ❌ 在主目錄直接進行開發（應該在 worktree 中）
