# Selenium 自動化測試

> ⚠️ **最高原則**：在使用者回覆 `OKOKYES` 確認測試計畫前，**絕對禁止**撰寫或執行任何測試程式碼！

---

## 觸發情境

此 Skill 通常由 CodeReview Skill 審查通過後串接觸發，此時 Claude 已知道：
- 剛剛開發了什麼功能
- 異動了哪些檔案
- 專案的技術架構

---

## 強制執行流程

```
Step 1: 回顧開發內容 → Step 2: 掃描程式碼 → Step 3: 產出測試計畫 → 等待 OKOKYES → Step 4: 執行測試
```

---

## Step 1: 回顧開發內容（自動執行）

> Claude 必須根據對話上下文，自動彙整以下資訊：

### 1.1 彙整開發資訊

從前面的對話中提取：

```
【開發內容回顧】

📋 開發功能：[從對話中取得，例：使用者登入功能]
📁 異動檔案：
   ├── [檔案1路徑]
   ├── [檔案2路徑]
   └── [檔案3路徑]

🎯 測試目標：
   - [功能1：例如登入成功]
   - [功能2：例如登入失敗顯示錯誤]
   - [功能3：例如記住我功能]
```

### 1.2 取得專案設定

**自動掃描設定檔取得測試 URL**：

```bash
# 掃描 application 設定
grep -rn 'server.port\|server.servlet.context-path' --include="*.properties" --include="*.yml" src/main/resources/

# 若找不到，預設為 http://localhost:8080
```

**輸出格式**：
```
🌐 測試環境：
   - Port: [從設定檔取得，例：8080]
   - Context Path: [從設定檔取得，例：/]
   - 測試 URL: http://localhost:[port][context-path]
```

> ⚠️ 若無法從設定檔取得，則詢問使用者確認測試 URL。

---

## Step 2: 掃描程式碼（必須完成）

> ⚠️ **禁止跳過此步驟**：必須掃描程式碼取得真實的元素定位，禁止猜測！

### 2.1 掃描前端元素

針對**異動的檔案及其相關頁面**進行掃描：

```bash
# 掃描 HTML/Thymeleaf 頁面的 ID 與 name 屬性
grep -rn 'id="\|name="\|th:id="\|th:name=' --include="*.html" src/main/resources/templates/

# 掃描表單元素
grep -rn '<form\|<input\|<button\|<select\|<textarea' --include="*.html" src/main/resources/templates/

# 掃描 JavaScript 的元素操作
grep -rn 'getElementById\|querySelector\|querySelectorAll' --include="*.js" src/main/resources/static/js/
```

### 2.2 掃描後端 API

```bash
# 掃描相關 Controller 端點
grep -rn '@GetMapping\|@PostMapping\|@PutMapping\|@DeleteMapping\|@RequestMapping' --include="*.java" src/main/java/

# 掃描登入相關（若測試涉及登入）
grep -rn 'login\|Login\|logout\|Logout\|session\|Session' --include="*.java" --include="*.html" src/
```

### 2.3 產出元素定位清單

掃描完成後，**必須**整理成以下格式：

```
【掃描結果 - 元素定位清單】

================================================================================
                              頁面元素掃描
================================================================================

## 頁面：[頁面名稱] (src/main/resources/templates/xxx.html)

| 元素名稱 | 定位方式 | 定位值 | 行號 | 用途說明 |
|---------|---------|--------|------|---------|
| 帳號輸入框 | id | username | :25 | 登入帳號輸入 |
| 密碼輸入框 | id | password | :30 | 登入密碼輸入 |
| 登入按鈕 | id | loginBtn | :35 | 提交登入表單 |
| 錯誤訊息 | id | errorMsg | :40 | 顯示登入錯誤 |

================================================================================
                              API 端點掃描
================================================================================

| 功能 | Method | Endpoint | Controller | 行號 |
|-----|--------|----------|------------|------|
| 登入頁面 | GET | /login | AuthController | :20 |
| 登入處理 | POST | /login | AuthController | :35 |
| 登出 | POST | /logout | AuthController | :50 |

================================================================================
```

> ⚠️ 根據 Step 1 的開發內容 + Step 2 的掃描結果，**自動**產出完整測試計畫。

### 3.1 測試計畫格式

```
【測試計畫書】

                              測試基本資訊
================================================================================

================================================================================
                              測試帳號需求
================================================================================

根據測試案例，需要以下帳號：

| # | 角色 | 用於測試案例 |
|---|-----|-------------|
| 1 | [角色名] | [TC-001, TC-002] |
```

### 3.2 測試案例設計原則

根據開發的功能，自動規劃以下類型的測試：

| 測試類型 | 說明 | 範例 |
|---------|------|------|
| **正向測試** | 正確輸入，預期成功 | 正確帳密登入成功 |
| **負向測試** | 錯誤輸入，預期失敗並顯示錯誤 | 錯誤密碼登入失敗 |
| **邊界測試** | 邊界值輸入 | 密碼長度剛好 8 碼 |
| **空值測試** | 必填欄位為空 | 帳號為空點擊登入 |
| **權限測試** | 驗證權限控制 | 未登入存取需授權頁面 |

### 3.3 等待使用者確認

> ⚠️ **強制等待**：必須等待使用者提供帳號密碼並回覆 `OKOKYES` 才能執行測試。

- ✅ 收到帳號密碼 + `OKOKYES` → 進入 Step 4
- ❌ 收到其他回覆 → 詢問需要調整什麼
- ❌ **絕對禁止**在未收到 `OKOKYES` 前撰寫任何測試程式碼

---

## Step 4: 執行測試

> 只有在收到 `OKOKYES` 確認後，才能進入此步驟。

### 4.1 Chrome 選項設定（強制）

```java
ChromeOptions options = new ChromeOptions();
options.addArguments("--no-sandbox");
options.addArguments("--disable-dev-shm-usage");
options.addArguments("--start-maximized");        // 視窗最大化
options.addArguments("--window-position=0,0");    // 視窗從左上角開始
// ❌ 禁止使用 --headless 參數！
```

### 4.2 撰寫測試程式碼

- 測試類別放在 `src/test/java/.../selenium/`
- 使用 `WebDriverWait` 等待元素（禁止 Thread.sleep）
- **所有元素定位必須來自 Step 2 的掃描結果**
- 每行程式碼加繁體中文註解

### 4.3 執行測試

```bash
# Java + Maven
source /home/ubuntu/selenium-env/bin/activate && ./mvnw test -Dtest=測試類別名稱

# Python + pytest
pytest tests/selenium/test_xxx.py -v
```

## 禁止事項（違反即為錯誤）

| 禁止行為 | 正確做法 |
|---------|---------|
| ❌ 不回顧開發內容就開始測試 | ✅ Step 1 先彙整開發資訊 |
| ❌ 未掃描程式碼就撰寫測試 | ✅ Step 2 完整掃描元素 |
| ❌ 未產出測試計畫就執行 | ✅ Step 3 產出計畫並等待確認 |
| ❌ 未收到 OKOKYES 就寫程式碼 | ✅ 等待明確的 OKOKYES |
| ❌ 猜測元素 ID | ✅ 從掃描結果取得真實 ID |
| ❌ 猜測測試網址 | ✅ 從設定檔取得或詢問使用者 |
| ❌ 使用 headless 模式 | ✅ 使用實體瀏覽器 |
| ❌ 使用 Thread.sleep | ✅ 使用 WebDriverWait |
| ❌ 跳過詢問測試帳號 | ✅ 必須詢問並取得帳號密碼 |

## 測試帳號處理規範

> ⚠️ **禁止在程式碼或文件中寫死帳號密碼**

1. 在 Step 3 測試計畫中列出所需帳號角色
2. 等待使用者提供帳號密碼
3. 使用者確認後，才在測試程式碼中使用（不 commit 帳密）
4. 執行完畢後提醒使用者帳密僅用於本次測試

### 測試失敗處理

若有測試失敗：
1. 列出失敗的測試案例與錯誤原因
2. 詢問使用者是否要修正問題
3. 若需修正，串接「開發」Skill
4. 修正後重新執行測試，直到全部通過
