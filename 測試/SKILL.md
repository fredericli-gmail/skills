# Selenium 自動化測試

> ⚠️ **最高原則**：在使用者回覆 `OKOKYES` 確認測試計畫前，**絕對禁止**撰寫或執行任何測試程式碼！

---

## 強制執行流程（必須按順序執行）

```
Phase 1: 資訊收集 → Phase 2: 程式碼掃描 → Phase 3: 測試計畫 → 等待 OKOKYES → Phase 4: 測試執行
```

---

## Phase 1: 資訊收集（必須完成）

> ⚠️ **禁止跳過此階段**：必須先確認測試目標才能繼續。

### 1.1 詢問測試目標

**必須向使用者確認以下資訊：**

```
【測試資訊確認】

請提供以下資訊，我才能開始規劃測試：

1. 【測試目標 URL】
   - 測試環境網址是？（例：http://localhost:8080）
   - 若有多個環境，請指定要測試哪一個

2. 【測試範圍】
   - 要測試哪些功能/頁面？
   - 是否需要登入？

3. 【專案路徑】（選填）
   - 專案的程式碼路徑？（預設：當前目錄）
   - 前端程式碼位置？（例：src/main/resources/templates/）
   - JavaScript 檔案位置？（例：src/main/resources/static/js/）

4. 【技術框架】
   - 前端框架？（Thymeleaf / React / Vue / 純 HTML）
   - 測試框架？（Java + Selenium / Python + pytest）
```

### 1.2 必須取得的資訊

| 項目 | 必要性 | 預設值 |
|-----|--------|--------|
| 測試目標 URL | **必須** | 無（必須由使用者提供） |
| 測試範圍 | **必須** | 無（必須由使用者提供） |
| 專案路徑 | 選填 | 當前目錄 |
| 技術框架 | 選填 | 自動偵測 |

> ❌ **禁止**：在未確認測試 URL 前就假設 `http://localhost:8080` 或其他網址！

## Phase 2: 程式碼掃描（必須完成）

> ⚠️ **禁止跳過此階段**：必須掃描程式碼取得真實的元素定位。

### 2.1 掃描前端元素

```bash
# 掃描 HTML/Thymeleaf 頁面的 ID 與 name 屬性
grep -rn 'id="\|name="\|th:id="\|th:name=' --include="*.html" [專案路徑]/

# 掃描表單元素
grep -rn '<form\|<input\|<button\|<select\|<textarea' --include="*.html" [專案路徑]/

# 掃描 JavaScript 的元素操作
grep -rn 'getElementById\|querySelector\|querySelectorAll\|By.ID\|By.CSS' --include="*.js" [專案路徑]/
```

### 2.2 掃描後端 API

```bash
# 掃描 Controller 端點
grep -rn '@GetMapping\|@PostMapping\|@PutMapping\|@DeleteMapping\|@RequestMapping' --include="*.java" [專案路徑]/

# 掃描登入相關
grep -rn 'login\|Login\|logout\|Logout\|auth\|Auth' --include="*.java" --include="*.html" [專案路徑]/
```

### 2.3 掃描設定檔

```bash
# 掃描 application 設定（確認 port、context-path）
grep -rn 'server.port\|server.servlet.context-path' --include="*.properties" --include="*.yml" [專案路徑]/
```

### 2.4 產出元素清單

掃描完成後，**必須**整理成以下格式：

```
【掃描結果 - 元素定位清單】

📁 專案路徑：[路徑]
🌐 偵測到的設定：
   - Port: [port]
   - Context Path: [path]
   - 建議測試 URL: [url]

---

## 頁面 1：[頁面名稱] ([檔案路徑])

| 元素名稱 | 定位方式 | 定位值 | 行號 | 用途說明 |
|---------|---------|--------|------|---------|
| 帳號輸入框 | id | username | :25 | 登入帳號輸入 |
| 密碼輸入框 | id | password | :30 | 登入密碼輸入 |
| 登入按鈕 | id | loginBtn | :35 | 提交登入表單 |

## 頁面 2：[頁面名稱] ([檔案路徑])
...

---

## API 端點清單

| 功能 | Method | Endpoint | Controller | 行號 |
|-----|--------|----------|------------|------|
| 登入 | POST | /api/login | AuthController | :42 |
...
```

## Phase 3: 測試計畫（必須完成）

> ⚠️ **禁止跳過此階段**：必須產出完整測試計畫並等待確認。

### 3.1 測試計畫格式

**必須**產出以下格式的測試計畫：

```
【測試計畫書】

================================================================================
                              測試基本資訊
================================================================================

📋 測試目標：[功能描述]
🌐 測試 URL：[確認過的 URL]
🖥️ 瀏覽器：Chrome（實體視窗，禁用 headless）
📂 測試類別路徑：[src/test/java/.../selenium/]

================================================================================
                              測試帳號需求
================================================================================

本次測試需要以下帳號：

| # | 角色 | 用途 | 帳號 | 密碼 |
|---|-----|------|------|------|
| 1 | [角色] | [用於哪個測試案例] | ❓待提供 | ❓待提供 |

【驗證機制】
- 圖形驗證碼：❓待確認取得方式
- OTP 驗證碼：❓待確認取得方式

================================================================================
                              測試案例清單
================================================================================

## TC-001: [測試案例名稱]

**測試目的**：[說明這個測試要驗證什麼]

**前置條件**：
1. [條件 1]
2. [條件 2]

**測試步驟**：

| 步驟 | 操作 | 元素定位 | 輸入/動作 | 預期結果 |
|-----|------|---------|----------|---------|
| 1 | 開啟頁面 | - | [URL] | 頁面載入完成 |
| 2 | 輸入帳號 | id=username | [帳號] | 欄位顯示輸入值 |
| 3 | 輸入密碼 | id=password | [密碼] | 欄位顯示遮罩 |
| 4 | 點擊登入 | id=loginBtn | click | - |
| 5 | 驗證結果 | - | - | 跳轉至首頁 |

**預期結果**：
- [具體的預期結果描述]

---

## TC-002: [測試案例名稱]
...

================================================================================
                              確認事項
================================================================================

請確認以下項目：

✅ 測試 URL 正確：[URL]
✅ 元素定位來源：從程式碼掃描取得
✅ 測試案例數量：[N] 個
✅ 需要帳號數量：[N] 組

❓ 待您提供：
1. 測試帳號密碼
2. 驗證碼取得方式（如有）

================================================================================

👉 確認無誤後，請回覆 OKOKYES 開始執行測試
   如需調整，請告知要修改的項目

================================================================================
```

### 3.2 等待確認

> ⚠️ **強制等待**：必須等待使用者回覆 `OKOKYES` 才能進入 Phase 4。

- ✅ 收到 `OKOKYES` → 進入 Phase 4
- ❌ 收到其他回覆 → 詢問需要調整什麼，修改計畫後再次等待確認
- ❌ **絕對禁止**在未收到 `OKOKYES` 前撰寫任何測試程式碼

---

## Phase 4: 測試執行

> 只有在收到 `OKOKYES` 確認後，才能進入此階段。

### 4.1 Chrome 選項設定（強制）

```java
ChromeOptions options = new ChromeOptions();
options.addArguments("--no-sandbox");
options.addArguments("--disable-dev-shm-usage");
options.addArguments("--start-maximized");        // 視窗最大化
options.addArguments("--window-position=0,0");    // 視窗從左上角開始
// ❌ 禁止使用 --headless 參數！
```

### 4.2 撰寫測試程式碼

- 測試類別放在 `src/test/java/.../selenium/`
- 使用 `WebDriverWait` 等待元素（禁止 Thread.sleep）
- 所有元素定位必須來自 Phase 2 的掃描結果

### 4.3 執行測試

```bash
# Java + Maven
source /home/ubuntu/selenium-env/bin/activate && ./mvnw test -Dtest=測試類別名稱

# Python + pytest
pytest tests/selenium/test_xxx.py -v
```

## 禁止事項（違反即為錯誤）

| 禁止行為 | 正確做法 |
|---------|---------|
| ❌ 未確認 URL 就開始測試 | ✅ Phase 1 詢問並確認 URL |
| ❌ 未掃描程式碼就撰寫測試 | ✅ Phase 2 完整掃描 |
| ❌ 未產出測試計畫就執行 | ✅ Phase 3 產出計畫並等待確認 |
| ❌ 未收到 OKOKYES 就寫程式碼 | ✅ 等待明確的 OKOKYES |
| ❌ 假設元素 ID | ✅ 從掃描結果取得真實 ID |
| ❌ 假設測試網址 | ✅ 由使用者明確提供 |
| ❌ 使用 headless 模式 | ✅ 使用實體瀏覽器 |
| ❌ 使用 Thread.sleep | ✅ 使用 WebDriverWait |

## 測試帳號取得流程

> ⚠️ **禁止在程式碼或文件中寫死帳號密碼**

在 Phase 3 測試計畫中列出所需帳號後，等待使用者提供：

1. **列出需求**：說明每個帳號的用途
2. **等待提供**：使用者提供帳號密碼
3. **確認完整**：確保所有需要的帳號都已取得
4. **收到 OKOKYES**：才開始撰寫測試

### 測試失敗處理

若有測試失敗：
1. 列出失敗的測試案例與錯誤原因
2. 詢問使用者是否要修正問題
3. 修正後重新執行測試，直到全部通過
